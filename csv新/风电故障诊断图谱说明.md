### 2\. 图谱Schema结构（给大模型识别用）

#### 2\.1 实体标签及含义

- Device：设备（风电相关设备，如变桨电动机、抱闸等，是故障发生的载体）

- FaultPhenomenon：故障现象（设备出现的具体故障表现，如抱闸无法打开，是用户提问中常见的核心诉求）

- FaultReason：故障原因（导致故障现象的具体诱因，如摩擦片法兰磨损，是故障诊断的核心逻辑节点）

- Measure：解决措施（针对故障原因的对应处理方法，是问答系统输出的关键解决方案）

#### 2\.2 关系类型及含义

- 表现为：Device（设备）→ FaultPhenomenon（故障现象），用于关联“设备”与“其出现的故障表现”，例如“抱闸设备表现为抱闸无法打开”。

- 原因是：FaultPhenomenon（故障现象）→ FaultReason（故障原因），用于关联“故障现象”与“其产生的具体原因”，例如“抱闸无法打开的原因是摩擦片法兰磨损”。

- 措施是：FaultReason（故障原因）→ Measure（解决措施），用于关联“故障原因”与“对应的解决方法”，形成完整的故障诊断\-解决方案闭环。

#### 2\.3 额外属性说明

所有关系均包含 source 属性，用于存储该条知识的来源文档名（如“1\.5MW风力发电机组直流串励变桨电动机故障原因分析\.docx”）。该属性可用于大模型回答用户提问时，标注知识来源，提升回答的可信度和可追溯性，符合故障诊断场景的严谨性要求。

以下模板可直接供大模型解析调用，替换占位符后即可查询Neo4j知识图谱，获取精准数据，无需额外修改语法：

- 查询某故障原因的解决措施及来源：MATCH \(r:FaultReason\{name:\&\#34;\{故障原因名\}\&\#34;\}\)\-\[m:措施是\]\-\&gt;\(n\) RETURN n\.name AS 解决措施, m\.source AS 知识来源;

- 查询某设备的所有故障现象：MATCH \(d:Device\{name:\&\#34;\{设备名\}\&\#34;\}\)\-\[p:表现为\]\-\&gt;\(n\) RETURN n\.name AS 故障现象;

- 查询所有带来源的知识（用于大模型批量检索、验证数据）：MATCH \(\)\-\[r\]\-\&gt;\(\) WHERE r\.source IS NOT NULL RETURN startNode\(r\)\.name AS 头实体, type\(r\) AS 关系, endNode\(r\)\.name AS 尾实体, r\.source AS 知识来源;

### 5\. 注意事项

- 图谱中实体名称、标签、关系的表述需保持一致（如“FaultPhenomenon”不可误写为“faultphenomenon”），避免大模型查询时出现语法错误或查询失败。

- 大模型调用Cypher模板时，需准确替换“\{故障现象名\}”“\{故障原因名\}”“\{设备名\}”等占位符，占位符替换需与图谱中实体名称完全匹配（包括大小写、空格）。

- 关系的source属性需与实际知识来源文档名完全一致，不可随意修改，确保大模型标注的来源可追溯、可验证。

- 后续新增故障知识时，需严格遵循“实体标签\-关系\-实体”的Schema结构，确保新增数据与现有图谱兼容，避免破坏大模型查询逻辑。
